# v2.11 User Service Upgrade — Mobile Number Validation for Kenya

## Context

CRS v2.11 ([release notes](https://github.com/egovernments/Citizen-Complaint-Resolution-System/releases/tag/v2.11)) introduces configurable mobile number validation. The user service no longer hardcodes India's `+91` / 10-digit pattern — it reads validation rules from MDMS, enabling country-specific phone formats.

This document covers what needs to change for the **Nairobi City County (Nai Pepea)** deployment to support Kenya phone numbers (`+254`, 9 digits after prefix).

---

## 1. Update egov-user Docker Image

**Old image**: whatever is currently in `docker-compose.deploy.yaml`
**New image**: `egovio/egov-user:master-d69ce29`

This build adds:
- MDMS-based mobile number validation (country prefix + length + regex)
- DB migration included in the same tag (`egov-user-db:master-d69ce29`)

### Where to change

In the Docker Compose file (`docker-compose.deploy.yaml` or `docker-compose.egov-digit.yaml`), update:

```yaml
egov-user:
  image: egovio/egov-user:master-d69ce29   # was: egovio/egov-user:<old-tag>
```

Also update the DB migration init container if applicable:
```yaml
egov-user-db:
  image: egovio/egov-user-db:master-d69ce29
```

### Environment variable

The user service has a toggle for mobile validation. Ensure this is set:

```yaml
MOBILE_NUMBER_VALIDATION_WORKAROUND_ENABLED: "false"
```

Setting it to `"false"` means the service will use the MDMS-based validation (the new behavior). Setting it to `"true"` bypasses validation entirely.

## 2. Create MDMS Record: `common-masters.UserValidation`

### Schema

The schema `common-masters.UserValidation` must exist in MDMS v2 before data records can be created.

**Schema definition** (create via `/mdms-v2/v2/_schema/create`):

```json
{
  "RequestInfo": {
    "apiId": "Rainmaker",
    "msgId": "1234",
    "authToken": "<token>"
  },
  "SchemaDefinition": {
    "tenantId": "ke",
    "code": "common-masters.UserValidation",
    "description": "Mobile number validation rules per country",
    "definition": {
      "type": "object",
      "properties": {
        "fieldType": { "type": "string" },
        "default": { "type": "boolean" },
        "isActive": { "type": "boolean" },
        "attributes": {
          "type": "object",
          "properties": {
            "prefix": { "type": "string" }
          },
          "required": ["prefix"]
        },
        "rules": {
          "type": "object",
          "properties": {
            "pattern": { "type": "string" },
            "minLength": { "type": "integer" },
            "maxLength": { "type": "integer" },
            "errorMessage": { "type": "string" },
            "allowedStartingCharacters": {
              "type": "array",
              "items": { "type": "string" }
            }
          },
          "required": ["pattern", "minLength", "maxLength"]
        }
      },
      "required": ["fieldType", "default", "isActive", "attributes", "rules"]
    },
    "isActive": true
  }
}
```

### Data Record — Kenya (`+254`)

Create via `/mdms-v2/v2/_create/common-masters.UserValidation`:

```json
{
  "RequestInfo": {
    "apiId": "Rainmaker",
    "msgId": "1234",
    "authToken": "<token>"
  },
  "Mdms": {
    "tenantId": "ke",
    "schemaCode": "common-masters.UserValidation",
    "data": {
      "fieldType": "mobile",
      "default": true,
      "isActive": true,
      "attributes": {
        "prefix": "+254"
      },
      "rules": {
        "pattern": "^[17][0-9]{8}$",
        "minLength": 9,
        "maxLength": 9,
        "errorMessage": "CORE_COMMON_MOBILE_ERROR",
        "allowedStartingCharacters": ["1", "7"]
      }
    },
    "isActive": true
  }
}
```

**Kenya phone number format**:
- Country code: `+254`
- Local number: 9 digits
- Starts with `1` (landline) or `7` (mobile)
- Example: `+254 712 345 678` → local part `712345678`
- Regex: `^[17][0-9]{8}$`

## 3. Who Reads This Config

Three consumers read `common-masters.UserValidation`:

### 3a. egov-user (backend)

The user service (`egovio/egov-user:master-d69ce29`) validates phone numbers on user create/update against the MDMS pattern. If the pattern doesn't match, registration is rejected.

### 3b. Novu Bridge (notifications)

`backend/novu-bridge/src/main/java/org/egov/novubridge/service/MdmsServiceClient.java`

Reads `data.attributes.prefix` to prepend the country code when sending SMS/WhatsApp notifications. It looks for the record with `data.default = true`.

### 3c. PGR Frontend (UI form validation)

`frontend/micro-ui/web/micro-ui-internals/packages/modules/pgr/src/hooks/pgr/useMobileValidation.js`

The `useMobileValidation` hook fetches the config and applies:
- `prefix` → shown in the phone input field (e.g., "+254")
- `pattern` → client-side regex validation
- `minLength` / `maxLength` → input field constraints
- `allowedStartingCharacters` → restricts first digit
- `errorMessage` → localization key for error display

**Priority order in the hook**:
1. `window.globalConfigs.getConfig("CORE_MOBILE_CONFIGS")` (if set)
2. MDMS `common-masters.UserValidation` (fieldType = "mobile")
3. Hardcoded fallback: `+91`, `^[6-9][0-9]{9}$`, length 10

---

## 4. Frontend Config (Optional Override)

If you want to set the phone validation via `globalConfigs.js` instead of (or in addition to) MDMS, add to `public/globalConfigs.js`:

```javascript
var defined_configs = {
  // ... existing configs ...
  CORE_MOBILE_CONFIGS: {
    mobilePrefix: "+254",
    mobileNumberPattern: "^[17][0-9]{8}$",
    mobileNumberLength: 9,
    mobileNumberAllowedStartingCharacters: ["1", "7"],
    mobileNumberErrorMessage: "CORE_COMMON_MOBILE_ERROR"
  }
};
```

This takes priority over MDMS in the frontend, but does **not** affect backend validation (egov-user) or notifications (Novu Bridge). For consistency, always create the MDMS record.

## 5. Localization

Add error message localization for the validation error:

```json
{
  "code": "CORE_COMMON_MOBILE_ERROR",
  "message": "Please enter a valid Kenyan mobile number (9 digits starting with 1 or 7)",
  "module": "rainmaker-common",
  "locale": "en_IN"
}
```

Create via `/localization/messages/v1/_upsert`.

## 6. Verification Steps

After deploying:

1. **Check egov-user image**: `docker inspect egov-user --format '{{.Config.Image}}'` should show `egovio/egov-user:master-d69ce29`

2. **Verify MDMS record exists**:
   ```bash
   curl -s -X POST http://localhost:18000/mdms-v2/v2/_search \
     -H 'Content-Type: application/json' \
     -d '{"RequestInfo":{"apiId":"Rainmaker"},"MdmsCriteria":{"tenantId":"ke","schemaCode":"common-masters.UserValidation"}}' \
     | python3 -m json.tool
   ```

3. **Test user creation with Kenya number**:
   - Create an employee via HRMS with mobile `712345678` — should succeed
   - Try mobile `012345678` — should fail validation (doesn't start with 1 or 7)

4. **Test frontend**:
   - Go to employee create complaint page
   - Phone input should show `+254` prefix
   - Entering 8 digits should show error; 9 valid digits should pass
