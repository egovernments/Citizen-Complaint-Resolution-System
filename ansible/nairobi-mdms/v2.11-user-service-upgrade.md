# v2.11 User Service Upgrade — Mobile Number Validation for Kenya

## Context

CRS v2.11 ([release notes](https://github.com/egovernments/Citizen-Complaint-Resolution-System/releases/tag/v2.11)) introduces configurable mobile number validation. The user service no longer hardcodes India's `+91` / 10-digit pattern — it reads validation rules from MDMS, enabling country-specific phone formats.

This document covers what needs to change for the **Nairobi City County (Nai Pepea)** deployment to support Kenya phone numbers (`+254`, 9 digits after prefix).

> **Schema migration note**: The MDMS schema was renamed from `common-masters.UserValidation` to `common-masters.MobileNumberValidation` and the data structure was simplified. The old nested `attributes.prefix` / `rules.pattern` shape is replaced with flat `countryCode` / `mobileNumberRegex` fields.

---

## 1. Update egov-user Docker Image

**Old image**: whatever is currently in `docker-compose.deploy.yaml`
**New image**: `egovio/egov-user:master-e22c7c5`

This build adds:
- MDMS-based mobile number validation (country prefix + regex)
- DB migration included in the same tag (`egov-user-db:master-e22c7c5`)

### Where to change

In the Docker Compose file (`docker-compose.deploy.yaml` or `docker-compose.egov-digit.yaml`), update:

```yaml
egov-user:
  image: egovio/egov-user:master-e22c7c5   # was: egovio/egov-user:<old-tag>
```

Also update the DB migration init container if applicable:
```yaml
egov-user-db:
  image: egovio/egov-user-db:master-e22c7c5
```

### Environment variable

The user service has a toggle for mobile validation. Ensure this is set:

```yaml
MOBILE_NUMBER_VALIDATION_WORKAROUND_ENABLED: "false"
```

Setting it to `"false"` means the service will use the MDMS-based validation (the new behavior). Setting it to `"true"` bypasses validation entirely.

## 2. Create MDMS Record: `common-masters.MobileNumberValidation`

### Schema

The schema `common-masters.MobileNumberValidation` must exist in MDMS v2 before data records can be created.

**Schema definition** (create via `/mdms-v2/schema/v1/_create`):

```json
{
  "RequestInfo": {
    "apiId": "Rainmaker",
    "ver": "1",
    "authToken": "<token>"
  },
  "SchemaDefinition": {
    "tenantId": "ke",
    "code": "common-masters.MobileNumberValidation",
    "description": "Mobile Number Validation Configuration",
    "definition": {
      "$schema": "http://json-schema.org/draft-07/schema#",
      "title": "Mobile Number Validation",
      "type": "object",
      "required": ["countryCode", "mobileNumberRegex"],
      "x-unique": ["countryCode"],
      "properties": {
        "tenantId": { "type": "string" },
        "countryCode": { "type": "string" },
        "mobileNumberRegex": { "type": "string" },
        "default": { "type": "boolean", "default": false }
      },
      "x-ref-schema": [],
      "additionalProperties": false
    },
    "isActive": true
  }
}
```

### Data Record — Kenya (`+254`)

Create via `/mdms-v2/v2/_create/common-masters.MobileNumberValidation`:

```json
{
  "RequestInfo": {
    "apiId": "Rainmaker",
    "ver": "1",
    "authToken": "<token>"
  },
  "Mdms": {
    "tenantId": "ke",
    "schemaCode": "common-masters.MobileNumberValidation",
    "data": {
      "countryCode": "+254",
      "mobileNumberRegex": "^[17][0-9]{8}$",
      "default": true
    },
    "isActive": true
  }
}
```

**Kenya phone number format**:
- Country code: `+254`
- Local number: 9 digits
- Starts with `1` (landline) or `7` (mobile)
- Example: `+254 712 345 678` → local part `712345678`
- Regex: `^[17][0-9]{8}$`

## 3. Who Reads This Config

Three consumers read `common-masters.MobileNumberValidation`:

### 3a. egov-user (backend)

The user service (`egovio/egov-user:master-e22c7c5`) validates phone numbers on user create/update against the MDMS `mobileNumberRegex`. If the pattern doesn't match, registration is rejected.

### 3b. Novu Bridge (notifications)

`backend/novu-bridge/src/main/java/org/egov/novubridge/service/MdmsServiceClient.java`

Reads `data.countryCode` to prepend the country code when sending SMS/WhatsApp notifications. It looks for the record with `data.default = true`.

### 3c. PGR Frontend (UI form validation)

`frontend/micro-ui/web/micro-ui-internals/packages/modules/pgr/src/hooks/pgr/useMobileValidation.js`

The `useMobileValidation` hook fetches the config and applies:
- `countryCode` → shown in the phone input field (e.g., "+254")
- `mobileNumberRegex` → client-side regex validation

**Priority order in the hook**:
1. `window.globalConfigs.getConfig("CORE_MOBILE_CONFIGS")` (if set)
2. MDMS `common-masters.MobileNumberValidation` (record with `default: true`)
3. Hardcoded fallback: `+91`, `^[6-9][0-9]{9}$`

---

## 4. Frontend Config (Optional Override)

If you want to set the phone validation via `globalConfigs.js` instead of (or in addition to) MDMS, add to `public/globalConfigs.js`:

```javascript
var defined_configs = {
  // ... existing configs ...
  CORE_MOBILE_CONFIGS: {
    countryCode: "+254",
    mobileNumberRegex: "^[17][0-9]{8}$"
  }
};
```

This takes priority over MDMS in the frontend, but does **not** affect backend validation (egov-user) or notifications (Novu Bridge). For consistency, always create the MDMS record.

## 6. Verification Steps

After deploying:

1. **Check egov-user image**: `docker inspect egov-user --format '{{.Config.Image}}'` should show `egovio/egov-user:master-e22c7c5`

2. **Verify MDMS record exists**:
   ```bash
   curl -s -X POST http://localhost:18000/egov-mdms-service/v2/_search \
     -H 'Content-Type: application/json' \
     -d '{"RequestInfo":{"apiId":"Rainmaker","ver":"1"},"MdmsCriteria":{"tenantId":"ke","schemaCode":"common-masters.MobileNumberValidation"}}' \
     | python3 -m json.tool
   ```

3. **Test user creation with Kenya number**:
   - Create an employee via HRMS with mobile `712345678` — should succeed
   - Try mobile `012345678` — should fail validation (doesn't start with 1 or 7)

4. **Test frontend**:
   - Go to employee create complaint page
   - Phone input should show `+254` prefix
   - Entering 8 digits should show error; 9 valid digits should pass
